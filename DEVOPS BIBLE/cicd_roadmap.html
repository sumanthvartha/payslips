01

What Is CI/CD & Why It Exists

Before CI/CD existed, teams would write code for weeks, then one day manually copy files to a server, run commands by hand, and pray it didn't break. If something broke, nobody knew which change caused it. Fixing it took hours. This was called "deployment day" and developers feared it.

CI/CD solves this by making deployment automatic, frequent, and safe.

CI — Continuous Integration

Every time a developer pushes code, the system automatically integrates it with everyone else's code, runs tests, and tells you immediately if something broke. You catch problems within minutes, not weeks.

💡

Key idea: Integrate code changes small and often rather than in one giant batch. The word "continuous" means this happens on every single push — not once a week.

CD — Continuous Delivery vs Continuous Deployment

People confuse these two. They are different:

TermMeaningHuman involved?
Continuous DeliveryCode is always in a deployable state. You can deploy anytime by clicking a button.Yes — human approves the final push
Continuous DeploymentEvery change that passes tests is automatically deployed to production. No human needed.No — fully automated

Most companies use Continuous Delivery for production (with a human approval gate) and Continuous Deployment for dev/staging environments.

The Real-World Problem CI/CD Solves

Without CI/CDWith CI/CD
Deploy once a month, big scary releaseDeploy 10-50 times a day safely
Bugs found weeks after writingBugs found within minutes of push
"It works on my machine"Tested in a clean environment every time
Manual steps, human errorsAutomated, consistent, repeatable
1 week to roll back a bad releaseRollback in 2 minutes

Core Concepts You Must Know

Pipeline

A series of automated stages that your code passes through from push to production. Each stage must pass before the next one starts. Think of it as an assembly line with quality checks.

Stage / Job / Step

TermWhat it isExample
PipelineThe whole processEverything from push to deploy
StageA group of related jobs"Test stage", "Build stage", "Deploy stage"
JobOne unit of work that runs on one machine"Run unit tests", "Build Docker image"
StepA single command inside a job"npm install", "docker build"

Runner / Agent

The machine that runs your pipeline. When you push code, GitHub spins up a fresh Linux/Windows/Mac machine, runs your pipeline steps on it, and destroys it when done. You never manage this machine yourself with GitHub Actions. With Jenkins, you manage your own machines (called agents).

Artifact

The output of the build stage. Could be a Docker image, a .zip file, a .jar file. This artifact is what gets deployed — not the raw source code.

Trigger

What starts the pipeline. Common triggers:

Push to main branch Pull Request opened Scheduled (every night at 2am) Manual button click Another pipeline finishes

Branch Strategy

Your pipeline behaves differently depending on which branch code is pushed to. The most common pattern:

BranchPipeline behavior
feature/*Run tests only — no deploy
developRun tests + deploy to DEV environment
mainRun tests + deploy to STAGING → wait for approval → deploy PRODUCTION

Docker & Containers (The Backbone)

Almost every modern CI/CD pipeline uses Docker. You must understand this.

Why Docker in CI/CD?

Your app needs specific software to run (Python 3.11, certain libraries, etc.). Docker packages your app + everything it needs into one container image. That image runs identically on your laptop, on the pipeline runner, and on AWS. No more "works on my machine."

Dockerfile — What It Is

A text file that tells Docker how to build your image. Every line is a layer:

# Start from official Python image (base layer)
FROM python:3.11-slim

# Set working directory inside container
WORKDIR /app

# Copy requirements first (for caching)
COPY requirements.txt .

# Install dependencies
RUN pip install -r requirements.txt

# Copy rest of your app code
COPY . .

# Tell Docker what port your app uses
EXPOSE 8000

# Command to start your app
CMD ["python", "app.py"]

Docker Image vs Container

TermAnalogyReality
Docker ImageA cake recipe + all ingredients boxed upRead-only snapshot of your app + dependencies
Docker ContainerThe actual cake being baked and servedA running instance of an image

Your CI/CD pipeline builds the image and stores it in ECR. AWS ECS then runs containers from that image.

Image Tagging Strategy

Every image needs a unique tag so you know exactly what version is running:

# Bad — "latest" tag makes rollback impossible
my-app:latest

# Good — use git commit SHA for exact traceability
my-app:a3f8c12

# Also good — semantic versioning
my-app:v1.4.2

# Best practice in pipelines — combine both
my-app:v1.4.2-a3f8c12

AWS Infrastructure Setup (One-Time)

Before your pipeline can run, these AWS resources must exist. You create them once and the pipeline uses them forever.

Step 1 — Create ECR Repository

ECR is AWS's private Docker registry. Your pipeline pushes images here.

# AWS CLI command to create ECR repo
aws ecr create-repository \
  --repository-name my-app \
  --image-scanning-configuration scanOnPush=true \
  --region us-east-1

# Result: you get a URL like:
123456789.dkr.ecr.us-east-1.amazonaws.com/my-app
💡

scanOnPush=true tells ECR to automatically scan your Docker image for security vulnerabilities every time you push it. Free and worth enabling.

Step 2 — Create ECS Cluster

ECS is where your containers run. Three things to create:

1

ECS Cluster

The container group. Think of it as your data center. Name: my-cluster

2

Task Definition

Blueprint for your container. Specifies: which ECR image to use, CPU (e.g. 256 units), RAM (e.g. 512 MB), port mappings, environment variables, and IAM role.

3

ECS Service

Keeps N copies of your container running. If one crashes, it restarts automatically. Name: my-service. This is what the pipeline updates during deployment.

Step 3 — Create IAM User for Pipeline

Your pipeline needs AWS credentials to push images and trigger deployments. Create a dedicated IAM user with minimum permissions:

# Create user
aws iam create-user --user-name github-actions-deployer

# Attach only needed policies (least privilege)
aws iam attach-user-policy \
  --user-name github-actions-deployer \
  --policy-arn arn:aws:iam::aws:policy/AmazonEC2ContainerRegistryPowerUser

aws iam attach-user-policy \
  --user-name github-actions-deployer \
  --policy-arn arn:aws:iam::aws:policy/AmazonECS_FullAccess

# Create access keys — save these for GitHub Secrets
aws iam create-access-key --user-name github-actions-deployer
⚠️

Never give this user AdministratorAccess. Only give it exactly the permissions it needs. This is called the Principle of Least Privilege.

Step 4 — Store Credentials in GitHub Secrets

GitHub Repo → Settings → Secrets and Variables → Actions → New repository secret

Secret NameValue
AWS_ACCESS_KEY_IDFrom IAM user access keys
AWS_SECRET_ACCESS_KEYFrom IAM user access keys
AWS_REGIONus-east-1 (or your region)
ECR_REPOSITORY123456789.dkr.ecr.us-east-1.amazonaws.com/my-app

Step 5 — VPC, Subnets & Security Groups for ECS

ECS containers run inside your VPC. You need:

A VPC (default VPC works for learning) 2+ subnets across availability zones Security group allowing inbound port 80/443 Application Load Balancer (for production)

Stage 1 — Source (Trigger)

The pipeline starts when code is pushed. The runner clones the exact commit that triggered the pipeline. Every subsequent stage works on that exact snapshot of code.

Stage 2 — Lint / Code Quality

Before even running tests, check that the code is formatted correctly and has no obvious errors. Linting catches syntax errors, unused variables, style violations. Fast and cheap to run.

# Python example
- name: Lint with flake8
  run: |
    pip install flake8
    flake8 . --max-line-length=120 --exclude=venv/

# JavaScript example
- name: Lint with ESLint
  run: npx eslint src/ --ext .js,.jsx

Stage 3 — Testing (Most Important Stage)

Multiple types of tests run here:

Test TypeWhat it checksSpeedRun in pipeline?
Unit testsIndividual functions in isolationVery fast (seconds)Always
Integration testsMultiple components working togetherMedium (minutes)Always
End-to-end (E2E)Full app flow like a real user (Selenium, Playwright)Slow (10-30 min)On staging only
Performance testsLoad testing, response timesSlowOn staging only
# Python pytest with coverage
- name: Run tests with coverage
  run: |
    pip install pytest pytest-cov
    pytest tests/ --cov=src --cov-report=xml --cov-fail-under=80

# --cov-fail-under=80 means pipeline FAILS if coverage drops below 80%
💡

Code coverage = percentage of your code that tests actually execute. If it's below your threshold, the pipeline fails. This forces developers to write tests.

Stage 4 — Security Scanning

Two types of security scans should run in every pipeline:

Scan TypeToolWhat it finds
SAST (Static Analysis)Bandit (Python), SonarQubeInsecure code patterns (SQL injection, hardcoded passwords)
Dependency scanSnyk, OWASP Dependency-CheckKnown CVEs in your libraries (e.g. log4shell)
Container scanTrivy, ECR built-in scannerVulnerabilities in OS packages inside your Docker image
IaC scanCheckov, tfsecSecurity misconfigurations in Terraform/CloudFormation
# Scan Docker image with Trivy
- name: Scan Docker image for vulnerabilities
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: '${{ env.ECR_URL }}:${{ github.sha }}'
    format: 'table'
    exit-code: '1'           # Pipeline FAILS if CRITICAL vulnerabilities found
    severity: 'CRITICAL,HIGH'

Stage 5 — Build (Create the Artifact)

The code has passed all checks. Now build the Docker image. Use the git commit SHA as the image tag — this gives you full traceability.

- name: Build Docker image
  run: |
    # Tag with both commit SHA and "latest"
    docker build \
      -t $ECR_URL:${{ github.sha }} \
      -t $ECR_URL:latest \
      --build-arg BUILD_DATE=$(date -u +%Y-%m-%dT%H:%M:%SZ) \
      --build-arg VERSION=${{ github.sha }} \
      .

Stage 6 — Push to Registry

# Login to ECR first
- name: Login to ECR
  run: |
    aws ecr get-login-password --region $AWS_REGION | \
    docker login --username AWS --password-stdin $ECR_URL

# Push image to ECR
- name: Push image
  run: |
    docker push $ECR_URL:${{ github.sha }}
    docker push $ECR_URL:latest

GitHub Actions — Deep Dive

File Structure

your-repo/
  ├── src/
  ├── tests/
  ├── Dockerfile
  └── .github/
       └── workflows/
            ├── ci.yml          # Runs on every PR — tests only
            ├── deploy-dev.yml  # Deploys to dev on push to develop
            └── deploy-prod.yml # Deploys to prod on push to main

You can have multiple workflow files. Each handles a different scenario.

Workflow Syntax — Every Keyword Explained

# ── Name shown in GitHub UI ──
name: CI Pipeline

# ── When to run this workflow ──
on:
  push:
    branches: [main, develop]     # On push to these branches
  pull_request:
    branches: [main]              # On PR targeting main
  schedule:
    - cron: '0 2 * * *'          # Every day at 2 AM UTC
  workflow_dispatch:              # Allow manual trigger from UI

# ── Environment variables available to ALL jobs ──
env:
  AWS_REGION: us-east-1
  ECR_URL: 123456789.dkr.ecr.us-east-1.amazonaws.com/my-app

# ── Jobs run in parallel by default ──
jobs:

  # Job 1
  test:
    runs-on: ubuntu-latest        # Machine type to use
    
    strategy:
      matrix:
        python-version: [3.10, 3.11]  # Run on multiple Python versions!
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Python
        uses: actions/setup-python@v4
        with:
          python-version: ${{ matrix.python-version }}
      
      - name: Cache dependencies    # Speed up pipeline with caching
        uses: actions/cache@v3
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-pip-${{ hashFiles('requirements.txt') }}
      
      - name: Install deps
        run: pip install -r requirements.txt
      
      - name: Run tests
        run: pytest tests/ --cov=src

  # Job 2 — only runs after "test" job passes
  build:
    needs: test                   # Dependency on previous job
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'  # Only on main branch
    
    outputs:                      # Share data between jobs
      image-tag: ${{ steps.build.outputs.tag }}
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Build
        id: build
        run: |
          docker build -t $ECR_URL:${{ github.sha }} .
          echo "tag=${{ github.sha }}" >> $GITHUB_OUTPUT

Reusable Workflows

If multiple pipelines share the same steps (like the deploy step), extract them into a reusable workflow and call it from others. Don't repeat yourself.

# .github/workflows/reusable-deploy.yml
on:
  workflow_call:                  # Makes this workflow reusable
    inputs:
      environment:
        required: true
        type: string
    secrets:
      AWS_ACCESS_KEY_ID:
        required: true

# Call it from another workflow:
jobs:
  deploy:
    uses: ./.github/workflows/reusable-deploy.yml
    with:
      environment: production
    secrets:
      AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}

AWS OIDC — The Better Way (No Long-Lived Credentials)

Storing AWS Access Keys as GitHub Secrets is okay but there's a better, more secure approach called OIDC (OpenID Connect).

Instead of storing permanent AWS credentials, you configure AWS to trust GitHub's identity system. GitHub generates a short-lived token for each pipeline run. No permanent keys stored anywhere.

# With OIDC — no secrets needed in GitHub!
- name: Configure AWS credentials
  uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: arn:aws:iam::123456789:role/GitHubActionsRole
    aws-region: us-east-1
    # GitHub automatically handles the token exchange
✅

OIDC is the production best practice. Tokens expire after 1 hour. Even if stolen, they're useless quickly. No long-lived credentials anywhere.

AWS Secrets Manager for App Secrets

Your application itself needs secrets (database passwords, API keys). Don't put these in environment variables hardcoded in ECS task definitions. Use AWS Secrets Manager:

# In your ECS Task Definition, reference secrets from Secrets Manager
{
  "secrets": [
    {
      "name": "DATABASE_PASSWORD",
      "valueFrom": "arn:aws:secretsmanager:us-east-1:123:secret:prod/db-pass"
    }
  ]
}
# ECS injects this as an environment variable at runtime
# The actual value is never in your code or pipeline YAML

Deployment Strategies

How you replace the old version with the new one matters enormously for uptime. There are four main strategies:

1. Recreate (Basic — Avoid in Production)

Stop all old containers. Start all new ones. There is downtime during the switch.

[Old v1] [Old v1] [Old v1]
         STOP ALL
         ↓ (downtime gap)
[New v2] [New v2] [New v2]
         START ALL

❌ Users get errors during deployment. Only use for non-critical apps or dev environments.

2. Rolling Update (ECS Default)

Replace old containers one by one. At any moment, some users hit old version, some hit new.

[Old v1] [Old v1] [Old v1]
         ↓ (rolling)
[New v2] [Old v1] [Old v1]
[New v2] [New v2] [Old v1]
[New v2] [New v2] [New v2]
💡

✅ No downtime. But v1 and v2 run simultaneously for a while — your API must handle both versions gracefully. ECS does this automatically with --force-new-deployment.

# ECS rolling update configuration in Task Definition
{
  "deploymentConfiguration": {
    "minimumHealthyPercent": 50,   # Keep at least 50% running during deploy
    "maximumPercent": 200          # Can briefly run 200% capacity
  }
}

3. Blue/Green Deployment (Gold Standard)

Run two identical environments: Blue (current live) and Green (new version). Switch traffic instantly. If green has problems, switch back to blue instantly.

Load Balancer
      │
      ├── Blue (v1) ← 100% traffic (LIVE)
      └── Green (v2) ← 0% traffic (NEW, ready)

      ↓ After health checks pass:

Load Balancer
      ├── Blue (v1) ← 0% traffic (kept for rollback)
      └── Green (v2) ← 100% traffic (NOW LIVE)
✅

Zero downtime. Instant rollback if something goes wrong — just flip traffic back to Blue. AWS CodeDeploy + ECS supports this natively.

# GitHub Actions — deploy with CodeDeploy Blue/Green
- name: Deploy Blue/Green via CodeDeploy
  run: |
    aws deploy create-deployment \
      --application-name my-app \
      --deployment-group-name my-deployment-group \
      --revision revisionType=AppSpecContent,content="..." \
      --deployment-config-name CodeDeployDefault.ECSAllAtOnce

4. Canary Deployment

Send a small % of real traffic to the new version. Watch for errors. If stable, gradually increase to 100%.

Step 1:  95% → v1  |  5% → v2 (canary)
         Wait 10 min, watch error rates
         
Step 2:  50% → v1  |  50% → v2
         Wait 10 min
         
Step 3:  0% → v1   |  100% → v2
         Decommission v1

AWS App Mesh or AWS CodeDeploy Canary can automate this. If error rate spikes at any step, it automatically stops and rolls back.

Choosing the Right Strategy

StrategyComplexityDowntimeRollback speedUse for
RecreateLowYesSlowDev env, non-critical
RollingMediumNoMediumMost apps
Blue/GreenHighNoInstantProduction APIs
CanaryHighNoAutomaticHigh-traffic production

Rollback Strategy

Bad deployments happen even with good pipelines. You need a plan to get back quickly.

Method 1 — Re-deploy Previous Image Tag

Since you tagged every image with a git commit SHA, you always know which image is which version. Rolling back = deploying the previous image tag.

# Emergency rollback — deploy previous commit's image
aws ecs update-service \
  --cluster prod-cluster \
  --service prod-service \
  --task-definition prod-task:42    # previous task definition revision
  --force-new-deployment

Method 2 — Automated Health Check Rollback

Configure ECS to automatically roll back if the new deployment fails health checks:

# In ECS Service configuration:
{
  "deploymentConfiguration": {
    "deploymentCircuitBreaker": {
      "enable": true,
      "rollback": true     # Auto rollback if health checks fail!
    }
  }
}
✅

Enable this always. If 3 consecutive health checks fail on the new containers, ECS automatically stops the deployment and rolls back to the previous version. No human needed.

Method 3 — Blue/Green Instant Rollback

If using Blue/Green deployment, just shift traffic back to Blue. Takes 10 seconds. This is why Blue/Green is preferred for critical services.

Rollback Pipeline

Create a dedicated manual rollback workflow in GitHub Actions that anyone can trigger in an emergency with a specific image tag:

name: Emergency Rollback

on:
  workflow_dispatch:            # Manual trigger from GitHub UI
    inputs:
      image_tag:
        description: 'Image tag to rollback to (e.g. a3f8c12)'
        required: true

jobs:
  rollback:
    runs-on: ubuntu-latest
    environment: production     # Still requires approval
    steps:
      - name: Rollback ECS to specific image
        run: |
          # Update task definition with old image tag
          aws ecs update-service \
            --cluster prod-cluster \
            --service prod-service \
            --force-new-deployment

Monitoring & Alerting

Deploying code is not the end. You need to know if your deployed app is actually healthy.

Pipeline Notifications

Your team needs to know when pipelines succeed or fail:

# Send Slack notification on pipeline failure
- name: Notify Slack on failure
  if: failure()                 # Only runs if a previous step failed
  uses: slackapi/slack-github-action@v1
  with:
    payload: |
      {
        "text": "❌ Deploy FAILED on ${{ github.ref }} by ${{ github.actor }}\n${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}"
      }
  env:
    SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK }}

# Send on success
- name: Notify Slack on success
  if: success()
  uses: slackapi/slack-github-action@v1
  with:
    payload: '{"text": "✅ Deploy to PRODUCTION succeeded"}'

AWS CloudWatch — Application Monitoring

After deployment, CloudWatch watches your app:

What to MonitorCloudWatch MetricAlert if...
Error rate5xx HTTP errorsSpikes after deployment
Response timeTarget response timeIncreases above threshold
CPU usageECS CPUUtilizationConsistently above 80%
MemoryECS MemoryUtilizationConsistently above 85%
Container healthECS running task countDrops below desired count
💡

Create CloudWatch Alarms for each metric. When an alarm triggers → SNS → Slack/email. This way you know about production problems before users complain.

Application Logs in CloudWatch

Configure your ECS task to send container logs to CloudWatch Logs automatically:

# In ECS Task Definition — log configuration
{
  "logConfiguration": {
    "logDriver": "awslogs",
    "options": {
      "awslogs-group": "/ecs/my-app",
      "awslogs-region": "us-east-1",
      "awslogs-stream-prefix": "ecs"
    }
  }
}

Now every print() / console.log() from your app appears in CloudWatch Logs. You can search across all container instances in one place.

Health Check Endpoint

Your app should have a /health endpoint that returns 200 OK when healthy. ECS and load balancers hit this to decide if your container should receive traffic:

# Simple Python Flask health endpoint
@app.route('/health')
def health():
    return {"status": "healthy", "version": os.getenv("VERSION")}, 200

Infrastructure as Code (IaC)

In everything above, I said "go to AWS Console and click to create ECR, ECS, etc." That works for learning, but real teams never click the console. Instead they write code to define infrastructure.

Why IaC?

Manual Console ClicksInfrastructure as Code
Can't reproduce exactly — "what did I configure last month?"Entire setup is in version-controlled files
Creating dev/staging/prod copies is painfulRun same code 3 times = 3 identical environments
One person knows how it's set upWhole team can read and change it
Disaster recovery is hardRebuild entire infrastructure in 10 minutes

Terraform — The Most Common IaC Tool

Terraform lets you define AWS infrastructure in .tf files. It figures out what to create, update, or delete.

# main.tf — Create ECR repo with Terraform
resource "aws_ecr_repository" "my_app" {
  name                 = "my-app"
  image_tag_mutability = "IMMUTABLE"    # Tags can't be overwritten

  image_scanning_configuration {
    scan_on_push = true
  }
}

# Create ECS Cluster
resource "aws_ecs_cluster" "main" {
  name = "my-cluster"
}

# Output the ECR URL for use in pipelines
output "ecr_url" {
  value = aws_ecr_repository.my_app.repository_url

DevSecOps — Security Built Into the Pipeline

Security is not a separate step at the end. It is built into every stage of the pipeline. This approach is called DevSecOps or "shift left security" (move security checks earlier in the process).

Security at Each Stage

StageSecurity CheckTool
Code pushPre-commit hooks — block secrets being committedgit-secrets, detect-secrets
Lint stageSAST — find insecure code patternsBandit (Python), Semgrep
Test stageDependency vulnerability scanSnyk, OWASP Dependency-Check
Build stageDocker image vulnerability scanTrivy, AWS ECR scanning
IaC stageMisconfiguration scanCheckov, tfsec
RuntimeAnomaly detection, runtime threatsAWS GuardDuty

Pre-commit Hook — Prevent Secrets in Code

# Install on developer machine — runs BEFORE every commit
pip install pre-commit detect-secrets

# .pre-commit-config.yaml
repos:
  - repo: https://github.com/Yelp/detect-secrets
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']
  
  - repo: https://github.com/pre-commit/pre-commit-hooks
    hooks:
      - id: check-yaml
      - id: end-of-file-fixer
      - id: trailing-whitespace
🚨

If a developer accidentally commits an AWS key or database password, it's in git history forever even if deleted later. Pre-commit hooks prevent this from happening at all.

AWS GuardDuty — Runtime Protection

GuardDuty continuously monitors your AWS account for suspicious activity — unusual API calls, data exfiltration attempts, compromised EC2 instances. Enable it with one click in AWS Console. It runs passively and alerts you if anything looks malicious.

Complete Real-World Example — End to End

Let's put everything together. A Python web app. GitHub → GitHub Actions → ECR → ECS. Full production-grade pipeline.

Project Structure

my-python-app/
  ├── src/
  │    └── app.py
  ├── tests/
  │    └── test_app.py
  ├── terraform/
  │    ├── main.tf          # ECR, ECS, IAM, VPC
  │    └── variables.tf
  ├── Dockerfile
  ├── requirements.txt
  └── .github/
       └── workflows/
            ├── ci.yml       # PRs — test only
            └── cd.yml       # main branch — full deploy

ci.yml — Runs on Every Pull Request

name: CI — Test on Pull Request

on:
  pull_request:
    branches: [main, develop]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.11'

      - name: Cache pip
        uses: actions/cache@v3
        with:
          path: ~/.cache/pip
          key: pip-${{ hashFiles('requirements.txt') }}

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Lint with flake8
        run: |
          pip install flake8
          flake8 src/ tests/

      - name: Security scan with bandit
        run: |
          pip install bandit
          bandit -r src/ -ll

      - name: Run tests with coverage
        run: |
          pytest tests/ --cov=src --cov-report=xml --cov-fail-under=80

      - name: Scan dependencies
        uses: snyk/actions/python@master
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}

cd.yml — Full Deploy on Push to Main

name: CD — Build and Deploy

on:
  push:
    branches: [main]

env:
  AWS_REGION: us-east-1
  ECR_URL: 123456789.dkr.ecr.us-east-1.amazonaws.com/my-app

jobs:
  # ── Job 1: Test (same as CI) ──
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v4
        with: { python-version: '3.11' }
      - run: pip install -r requirements.txt
      - run: pytest tests/ --cov=src --cov-fail-under=80

  # ── Job 2: Build & Push Image ──
  build:
    needs: test
    runs-on: ubuntu-latest
    outputs:
      image-tag: ${{ github.sha }}

    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/GitHubActionsRole
          aws-region: ${{ env.AWS_REGION }}

      - name: Login to ECR
        uses: aws-actions/amazon-ecr-login@v2

      - name: Build Docker image
        run: docker build -t $ECR_URL:${{ github.sha }} .

      - name: Scan image with Trivy
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: '${{ env.ECR_URL }}:${{ github.sha }}'
          exit-code: '1'
          severity: 'CRITICAL'

      - name: Push to ECR
        run: docker push $ECR_URL:${{ github.sha }}

  # ── Job 3: Deploy to Staging ──
  deploy-staging:
    needs: build
    runs-on: ubuntu-latest
    environment: staging

    steps:
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/GitHubActionsRole
          aws-region: ${{ env.AWS_REGION }}

      - name: Update ECS task definition with new image
        id: task-def
        uses: aws-actions/amazon-ecs-render-task-definition@v1
        with:
          task-definition: staging-task-def.json
          container-name: my-app
          image: ${{ env.ECR_URL }}:${{ needs.build.outputs.image-tag }}

      - name: Deploy to staging ECS
        uses: aws-actions/amazon-ecs-deploy-task-definition@v1
        with:
          task-definition: ${{ steps.task-def.outputs.task-definition }}
          service: staging-service
          cluster: staging-cluster
          wait-for-service-stability: true   # Wait until healthy

      - name: Notify Slack — staging deployed
        uses: slackapi/slack-github-action@v1
        with:
          payload: '{"text": "✅ Staging deploy succeeded — awaiting prod approval"}'
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK }}

  # ── Job 4: Deploy to Production (requires approval) ──
  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production       # Pauses here for human approval in GitHub

    steps:
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/GitHubActionsRole
          aws-region: ${{ env.AWS_REGION }}

      - name: Update ECS task definition
        id: task-def
        uses: aws-actions/amazon-ecs-render-task-definition@v1
        with:
          task-definition: prod-task-def.json
          container-name: my-app
          image: ${{ env.ECR_URL }}:${{ needs.build.outputs.image-tag }}

      - name: Deploy to production ECS (Blue/Green)
        uses: aws-actions/amazon-ecs-deploy-task-definition@v1
        with:
          task-definition: ${{ steps.task-def.outputs.task-definition }}
          service: prod-service
          cluster: prod-cluster
          codedeploy-appspec: appspec.yml   # Blue/Green via CodeDeploy
          wait-for-service-stability: true

      - name: Notify Slack — production deployed
        if: success()
        uses: slackapi/slack-github-action@v1
        with:
          payload: '{"text": "🚀 Production deploy succeeded — ${{ github.sha }}"}'
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK }}

      - name: Notify Slack — production FAILED
        if: failure()
        uses: slackapi/slack-github-action@v1
        with:
          payload: '{"text": "🔥 Production deploy FAILED — ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}"}'
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK }}

The Full Picture — What Happens When You Push

1

git push to main

GitHub detects the push and reads .github/workflows/cd.yml

2

GitHub spins up a Linux machine

Fresh ubuntu-latest VM is provisioned in GitHub's cloud infrastructure.

3

Tests run

Lint, security scan, unit tests, coverage check. If any fail — pipeline stops. Nothing deploys.

4

Docker image built and scanned

Image built with commit SHA as tag. Trivy scans for vulnerabilities. If CRITICAL found — pipeline stops.

5

Image pushed to ECR

123456789.dkr.ecr.us-east-1.amazonaws.com/my-app:a3f8c12 is now stored in AWS.

6

Staging deployment

ECS task definition updated with new image. ECS pulls image from ECR. Rolling update starts. Pipeline waits until all staging containers are healthy.

7

Slack alert sent

"Staging deployed successfully — awaiting production approval."

8

Pipeline pauses — approval gate

GitHub sends email/Slack to approvers. Approver logs into GitHub, reviews staging, clicks "Approve". QA team tests staging for 30 minutes.

9

Production deployment

Blue/Green deployment via CodeDeploy. New containers (Green) start up. Load balancer switches traffic from Blue to Green. Health checks pass.

10

Slack success alert

"🚀 Production deploy succeeded — a3f8c12". CloudWatch monitoring your app for the next hour for any anomalies.

Dev → Staging → Production
Split your pipeline into stages. Create dev and staging environments. Add GitHub Environment protection rules and the manual approval gate.

The Full Package
Reusable workflows, canary deployments, AWS GuardDuty, Cost monitoring, multi-region deployments, disaster recovery runbooks. This is senior-level DevOps.

What is Version Control?
Why VCS exists, types of VCS, Git's place in history
Beginner

// Core Concepts

Version Control System (VCS)
A system that records changes to files over time so you can recall specific versions later. Think of it as a "time machine" for your code — you can go back to any point in history.
Types of VCS
Local VCS — Simple database of changes (RCS). All history on one machine.
Centralized VCS — Single server holds all versioned files (SVN, CVS). Everyone checks out from one place. Single point of failure.
Distributed VCS — Every client mirrors the entire repository including history (Git, Mercurial). No single point of failure.
Why Git Won
Speed, data integrity, support for distributed non-linear workflows, free and open source, and industry adoption. Created by Linus Torvalds in 2005 for Linux kernel development.
Git stores snapshots of your entire project, not just diffs between files. This is what makes it fast and reliable.

How Git Stores Data (Object Model)
Blobs, Trees, Commits, Tags — Git's internal storage
Beginner

// Git Object Types

The 4 Object Types
Blob — stores file content (no filename, no metadata)
Tree — stores directory structure (maps filenames to blobs/trees)
Commit — stores a snapshot pointer (tree + parent + author + message)
Tag — annotated label for a specific commit
Content-Addressable Storage
Every object is identified by a SHA-1 hash of its content. If two files have identical content, Git stores them only once. This is why Git is incredibly efficient.
# Inspect the .git directory ls .git/ # objects/ — all your blobs, trees, commits # refs/ — pointers to commits (branches, tags) # HEAD — pointer to current branch # See what a commit looks like internally git cat-file -t HEAD # type: commit git cat-file -p HEAD # content
Understanding Git's object model makes every Git command make sense. A branch is just a file containing a commit SHA. HEAD is a file pointing to a branch file.

✓
The Three Trees (Working Dir, Index, HEAD)
Git's most fundamental concept — the staging model
Core

// Git's Three Areas

Working Directory
Where you edit files. Git tracks changes here but doesn't version them yet. Files can be tracked (modified) or untracked (new).
Staging Area (Index)
A preparation area. You explicitly add changes here before committing. This lets you craft precise, logical commits from messy working changes.
Repository (HEAD)
The permanent, versioned history. Once committed, data is safe. HEAD always points to the latest commit of your current branch.
Working Directory → (git add) → Staging → (git commit) → HEAD HEAD → (git checkout) → Working Directory Staging → (git reset HEAD) → Working Directory
The staging area is Git's superpower. Other VCS go straight from working dir to committed. Git's index lets you build clean commits from messy work.

git status & git add

// Staging Changes

git commit

// Committing

# Commit staged changes git commit # Commit with inline message git commit -m "feat: add login button" # Stage all tracked files AND commit git commit -am "fix: correct typo in README" # Fix the last commit (message or files) git commit --amend git commit --amend -m "corrected message" # Create an empty commit (e.g., to trigger CI) git commit --allow-empty -m "trigger: pipeline"

type(scope): description

feat: new feature | fix: bug fix | docs: documentation
style: formatting | refactor: code restructure | test: tests
chore: build/tool changes | ci: CI config changes
A good commit message answers: "If applied, this commit will..." The subject line should be 50 chars or less, imperative mood: "Add feature" not "Added feature".
Never amend commits that have already been pushed to a shared remote. It rewrites history and causes problems for teammates.

Creating & Managing Branches
branch, checkout, switch — full branch lifecycle
Intermediate

// Branch Operations

# List branches (* = current) git branch git branch -a # include remote branches git branch -v # show last commit on each # Create a branch git branch feature/login # Switch to a branch git switch feature/login # modern (Git 2.23+) git checkout feature/login # classic # Create AND switch in one command git switch -c feature/login git checkout -b feature/login # Rename a branch git branch -m old-name new-name # Delete a branch (safe — only if merged) git branch -d feature/login # Force delete (unmerged) git branch -D feature/login
A branch is just a lightweight pointer (a 41-byte file with a SHA). Creating thousands of branches costs almost nothing.
✓
git merge (all strategies)
Fast-forward, 3-way merge, squash merge, no-ff
Intermediate

// Merge Strategies

Fast-Forward Merge
When the target branch hasn't diverged, Git simply moves the pointer forward. No merge commit created. Clean linear history.
3-Way Merge
When both branches have diverged, Git finds the common ancestor and creates a merge commit combining both histories.
# Switch to target branch first git switch main # Fast-forward merge (default when possible) git merge feature/login # Always create merge commit (no fast-forward) git merge --no-ff feature/login # Squash all feature commits into one git merge --squash feature/login git commit -m "feat: add login feature" # Abort a merge in progress git merge --abort
Use --no-ff in team settings to preserve the fact that a feature branch existed. Use --squash to keep main history clean with one commit per feature.
✓
Resolving Merge Conflicts
Understand conflict markers, resolve manually, use tools
Intermediate

// Conflict Resolution

Conflict Markers
<<<<<<< HEAD (your current branch) your version of the code ======= incoming version from merge >>>>>>> feature/login (incoming branch)
Resolution Steps
# 1. See which files have conflicts git status # 2. Open files and edit to resolve (remove markers) # 3. Mark as resolved git add resolved-file.txt # 4. Complete the merge git commit # Use a visual merge tool git mergetool # Use ours or theirs strategy for a whole file git checkout --ours filename.txt git checkout --theirs filename.txt
Always read both sides of a conflict before choosing. Blindly accepting "ours" or "theirs" can silently discard important changes.
✓
git rebase
Rebase vs merge, interactive rebase, golden rule
Intermediate

// Rebasing

What Rebase Does
Takes commits from your branch and re-applies them on top of another branch. Creates a linear history as if you started your work from the latest main, not from the point you branched.
# Rebase feature onto main git switch feature/login git rebase main # Interactive rebase — edit last 3 commits git rebase -i HEAD~3 # In the editor, use these commands: # pick — keep commit as-is # reword — keep commit, edit message # edit — pause to amend this commit # squash — combine with previous commit # fixup — squash, discard this message # drop — delete this commit entirely # Continue after resolving conflicts git rebase --continue # Abort a rebase git rebase --abort
GOLDEN RULE: Never rebase commits that exist on a public/shared remote branch. Rebase rewrites history — it changes SHA hashes. This breaks everyone else's work.
✓
git cherry-pick
Apply specific commits from one branch to another
Intermediate

// Cherry Picking

Use Case
You fixed a bug in a feature branch but need that fix on main right now. Cherry-pick copies just that commit.
# Apply a single commit to current branch git cherry-pick abc1234 # Apply a range of commits git cherry-pick abc1234^..def5678 # Cherry-pick without committing (stage only) git cherry-pick -n abc1234 # Cherry-pick with original author info git cherry-pick -x abc1234 # Abort if conflict git cherry-pick --abort
Cherry-pick creates a new commit with a different SHA but the same changes. Use it sparingly — it can lead to duplicate commits if the original branch is later merged.

git fetch, pull, push
Sync with remotes — the complete picture
Intermediate

// Remote Sync Operations

fetch vs pull
fetch — downloads remote changes but does NOT merge. Safe to run anytime. pull = fetch + merge (or rebase with --rebase flag). Can cause conflicts.
# Fetch all remotes git fetch git fetch origin git fetch --all # all remotes # Pull (fetch + merge) git pull git pull origin main # Pull with rebase instead of merge git pull --rebase # Push to remote git push origin main # Push and set upstream tracking git push -u origin feature/login # After -u, you can just use: git push # Force push (DANGEROUS — only on your own branches) git push --force-with-lease # safer than --force # Delete remote branch git push origin --delete feature/login
Always prefer `git pull --rebase` over plain `git pull`. It keeps your history linear and avoids unnecessary merge commits.
Never force push to main/master. Only use force push on your own feature branches, and prefer --force-with-lease which checks for new remote commits first.
✓
Tracking Branches & Upstream
Understand origin/main, remote-tracking refs
Intermediate

// Tracking Branches

Remote-Tracking Branches
Local read-only references that mirror remote branch state at last fetch. Named like `origin/main`. They're NOT the same as your local `main`.
# See all branches including remote-tracking git branch -a # See tracking relationships git branch -vv # Set upstream for existing branch git branch --set-upstream-to=origin/main main # Check what's ahead/behind remote git status # shows "ahead 2, behind 1" # Checkout a remote branch locally git checkout --track origin/feature-x # Or shorthand: git switch feature-x # auto-tracks if remote exists # Clean up stale remote-tracking refs git fetch --prune

Undoing Things & Recovery

reset, revert, restore, reflog — Git never truly loses data

0%
▾
✓
git restore & git checkout (files)
Discard working directory changes safely
Advanced

// Discarding Changes

# Discard unstaged changes in a file (modern) git restore filename.txt # Discard unstaged changes in ALL files git restore . # Unstage a file (keep working dir changes) git restore --staged filename.txt # Restore file from a specific commit git restore --source=HEAD~2 filename.txt # Classic equivalent (still works) git checkout -- filename.txt # Clean all untracked files git clean -fd # -f=force, -d=directories git clean -n # dry run first (shows what would be deleted)
git restore and git clean permanently destroy uncommitted changes. There's no undo. Run `git clean -n` first to preview.
✓
git reset (soft, mixed, hard)
Move HEAD and optionally unstage/discard changes
Advanced

// Reset Modes

The Three Modes
--soft: Move HEAD only. Changes stay staged.
--mixed (default): Move HEAD + unstage. Changes stay in working dir.
--hard: Move HEAD + unstage + discard changes. DESTRUCTIVE.
# Undo last commit, keep changes staged git reset --soft HEAD~1 # Undo last commit, keep changes in working dir (default) git reset HEAD~1 git reset --mixed HEAD~1 # Undo last commit, DISCARD all changes ⚠️ git reset --hard HEAD~1 # Reset to a specific commit git reset --hard abc1234 # Unstage a specific file (mixed reset on file) git reset HEAD filename.txt
--hard reset permanently destroys uncommitted changes. Combined with reflog, you can recover committed work — but not uncommitted work.
✓
git revert
Undo commits safely without rewriting history
Advanced

// Safe Undo with Revert

Revert vs Reset
Revert creates a NEW commit that undoes the changes of a previous commit. History is preserved. Safe for public/shared branches. Reset moves HEAD backwards — rewrites history.
# Revert the last commit git revert HEAD # Revert a specific commit git revert abc1234 # Revert without auto-committing (stage only) git revert -n abc1234 # Revert a range of commits (newest first) git revert HEAD~3..HEAD # Revert a merge commit (must specify parent) git revert -m 1 abc1234
Use revert on shared/public branches. Use reset only on private local branches. This is the professional rule every team should follow.
✓
git reflog — The Safety Net
Recover "lost" commits, branches, and resets
Advanced

// Reflog — Git's Secret Undo History

What is Reflog?
The reflog records every time HEAD moves — commits, resets, checkouts, rebases. It's your local safety net. Entries expire after 90 days by default.
# View reflog git reflog # Output looks like: # abc1234 HEAD@{0}: commit: add feature # def5678 HEAD@{1}: reset: moving to HEAD~2 # ghi9012 HEAD@{2}: checkout: moving from main to feature # Recover "lost" commits after a hard reset git reset --hard HEAD@{2} # Recover a deleted branch git reflog # find last commit SHA of deleted branch git branch recovered-branch abc1234 # View reflog for a specific branch git reflog feature/login
Before any scary Git operation (hard reset, force push, etc.) run `git reflog` first so you know where to return if things go wrong.
✓
git stash
Temporarily shelve work to switch context
Advanced

// Stashing

# Stash current changes git stash # Stash with a meaningful name git stash push -m "wip: login form validation" # Include untracked files git stash -u # List all stashes git stash list # Apply most recent stash (keep it in list) git stash apply # Apply and remove from list git stash pop # Apply a specific stash git stash apply stash@{2} # Delete a stash git stash drop stash@{0} # Clear all stashes git stash clear # Create a branch from a stash git stash branch feature/stashed-work stash@{0}

Branching Workflows (Git Flow, GitHub Flow)
Industry standard workflows for teams

// Workflow Strategies

GitHub Flow (simple — best for most teams)
1. main is always deployable
2. Create a descriptive feature branch from main
3. Add commits
4. Open a Pull Request
5. Review, discuss, iterate
6. Merge to main → deploy immediately
Git Flow (complex — for versioned releases)
main — production-ready code only
develop — integration branch
feature/* — new features (branch from develop)
release/* — release prep (from develop, merge to main+develop)
hotfix/* — urgent fixes (from main, merge to main+develop)
For DevOps/cloud projects, GitHub Flow or trunk-based development is preferred. Git Flow is complex and often overkill unless you have multiple production versions to support.

Git Hooks
Automate actions at key Git lifecycle events
Pro

// Hooks

What Are Hooks?
Scripts in .git/hooks/ that run automatically at Git lifecycle events. Use them to enforce standards, run tests, or lint code before commits.
# Key hook scripts (make them executable: chmod +x) # pre-commit → runs before commit, can abort it # commit-msg → validates commit message format # pre-push → runs before pushing # post-merge → runs after merging # post-checkout → runs after checkout # Example: pre-commit hook (.git/hooks/pre-commit) #!/bin/bash npm run lint if [ $? -ne 0 ]; then echo "Linting failed. Commit aborted." exit 1 fi # Share hooks via repo (hooks aren't committed by default) git config core.hooksPath .githooks # Put hooks in .githooks/ and commit that directory
For team-wide hooks, use tools like Husky (Node.js) or pre-commit (Python). These make hooks versioned and shareable across the team.

git submodule & subtree
Manage repos within repos
Pro

// Submodules

Submodule vs Subtree
Submodule: links to a specific commit of another repo. Keeps repos separate. More complex workflow. Subtree: merges another repo's history into your repo. Simpler to use, history is inline.
# Add a submodule git submodule add https://github.com/user/lib.git libs/lib # Clone a repo with submodules git clone --recurse-submodules https://github.com/user/repo.git # Initialize & update submodules after clone git submodule init git submodule update git submodule update --init --recursive # Update all submodules to latest git submodule update --remote
✓
git bisect
Binary search through history to find a bug's origin
Pro

// Bug Hunting with Bisect

How Bisect Works
Uses binary search to find which commit introduced a bug. You tell Git "good" and "bad" commits, and it checks out the midpoint for you to test, halving the search space each time.
# Start bisect session git bisect start # Mark current commit as bad git bisect bad # Mark a known-good commit git bisect good v1.0.0 # Git checks out midpoint — test your app, then: git bisect good # if bug not present git bisect bad # if bug is present # Repeat until Git finds the exact commit # Automated bisect with a test script git bisect run ./test-script.sh # End bisect session git bisect reset
Bisect can find a bug in 1000 commits in just 10 tests. With an automated test script, it runs completely unattended.

// Git + CI/CD

Key CI/CD Git Patterns
CI pipelines are triggered by Git events: push, pull_request, tag creation. Pipeline behavior should differ based on branch (feature → test only, main → test + deploy).
# GitHub Actions: trigger on push and PR on: push: branches: [ main, develop ] pull_request: branches: [ main ] # Trigger on tag push (for releases) push: tags: [ 'v*.*.*' ] # Common CI Git operations in pipeline scripts git log --oneline -5 # Show recent commits in build git describe --tags --always # Get version from nearest tag git rev-parse --short HEAD # Short SHA for image tagging git diff --name-only HEAD~1 # Changed files (conditional builds) # Fetch all branches in CI (shallow clones miss history) git fetch --unshallow
In CI, always use `git rev-parse --short HEAD` to tag Docker images and deployments. This creates a direct traceability link from running code back to the exact commit.

Git Performance & Large Files
Git LFS, gc, shallow clones, repo cleanup
Expert

// Performance

# Git LFS (Large File Storage) for binaries, videos, models git lfs install git lfs track "*.psd" "*.mp4" "*.bin" git add .gitattributes # Now normal git add/commit works for LFS files # Clean up repo and optimize storage git gc # garbage collect git gc --aggressive # thorough but slow # Find largest objects in repo git rev-list --objects --all | git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' | sort -k3 -n -r | head -20 # Prune unreachable objects git prune # Count objects git count-objects -vH
Never commit binary files (images, binaries, videos) directly to Git. Use Git LFS. Large binaries bloat the repo permanently — even after deletion, they stay in history.

// Worktrees

What is a Worktree?
Multiple working directories linked to the same Git repo. Switch between branches instantly without stashing. Each worktree has its own working dir and HEAD but shares the same .git object store.
# Add a worktree for a branch git worktree add ../feature-login feature/login # Create a new branch in a new worktree git worktree add -b hotfix/critical ../hotfix # List worktrees git worktree list # Remove a worktree git worktree remove ../feature-login # Prune stale worktree references git worktree prune
Worktrees are perfect when you need to fix a hotfix on main while in the middle of a large feature. No stashing or branch switching needed — both codebases open at once.
